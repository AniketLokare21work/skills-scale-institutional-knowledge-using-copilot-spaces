# OctoAcme — Role-Based Communication Checklist

## Purpose
Ensure all key personas are involved in communication planning and execution for each project phase. This checklist helps teams avoid communication gaps and ensures no critical stakeholder is left out.

---

## Project Initiation Phase — Communication Checklist

**When:** During project kickoff and business case development  
**Owner:** Project Manager

### Personas to Involve
- [ ] **Sponsor/Stakeholder** — Needs to approve problem statement and success metrics
- [ ] **Product Manager** — Defines the problem and success criteria
- [ ] **Project Manager** — Coordinates initiation and alignment
- [ ] **Tech Lead** — Provides technical feasibility input
- [ ] **Developers** — Early assessment of scope and approach
- [ ] **QA Lead** — Quality considerations for acceptance criteria
- [ ] **Security Officer** — Security and compliance implications (if applicable)

### Communication Actions
- [ ] Schedule kickoff meeting with all stakeholders
- [ ] Share Project One-pager draft with stakeholders before meeting
- [ ] Confirm Sponsor/Stakeholder alignment via email or meeting
- [ ] Document decision to move to planning
- [ ] Share approved charter with full team
- [ ] Send initiation summary to broader organization (if needed)

### Key Messages
- Problem statement and why this project matters
- Success metrics and expected outcomes
- Timeline and key milestones
- Roles and responsibilities
- Next steps and decision gates

---

## Project Planning Phase — Communication Checklist

**When:** During backlog refinement, estimation, and sprint planning  
**Owner:** Project Manager + Product Manager

### Personas to Involve
- [ ] **Product Manager** — Finalizes backlog priorities and acceptance criteria
- [ ] **Developers** — Estimate and refine technical approach
- [ ] **Tech Lead** — Reviews technical design and identifies risks
- [ ] **QA Lead** — Defines test strategy and acceptance criteria validation
- [ ] **Project Manager** — Coordinates timeline and milestones
- [ ] **Sponsor/Stakeholder** — Reviews plan and approves resources (if gate meeting needed)
- [ ] **Security Officer** — Identifies security requirements and test scenarios

### Communication Actions
- [ ] Share prioritized backlog with team
- [ ] Conduct planning meeting with all personas
- [ ] Share sprint/iteration plan with team and stakeholders
- [ ] Communicate Definition of Done criteria
- [ ] Share risk register and mitigation plans
- [ ] Confirm resource availability and team commitments
- [ ] Send planning summary to stakeholders

### Key Messages
- Scope and deliverables for upcoming sprint/iteration
- Timeline and release milestones
- Definition of Done criteria
- Top risks and mitigation plans
- Resource commitments and team roles

---

## Execution Phase — Communication Checklist

**When:** During sprint/iteration execution (ongoing)  
**Owner:** Project Manager + Developers

### Personas to Involve (Daily/Weekly)
- [ ] **Developers** — Daily standup to report progress
- [ ] **QA Lead** — Test status and defect tracking
- [ ] **Tech Lead** — Technical blockers and decisions
- [ ] **Project Manager** — Blocker escalation and coordination
- [ ] **Product Manager** — Clarifications on acceptance criteria

### Personas to Involve (Weekly)
- [ ] **Sponsor/Stakeholder** — Weekly or milestone status update
- [ ] **Security Officer** — Security review findings and remediation status (if applicable)
- [ ] **Communications Lead** — Status dashboard updates

### Communication Actions
- [ ] Conduct daily standups (15 min) with delivery team
- [ ] Hold weekly delivery sync with PM, PdM, Tech Lead, QA Lead
- [ ] Send weekly status update to Sponsor/Stakeholder (highlight progress, risks, blockers)
- [ ] Update project board and risk register
- [ ] Communicate blockers immediately when escalation needed
- [ ] Schedule demos/reviews with stakeholders
- [ ] Share test results and quality metrics weekly

### Key Messages
- Progress toward sprint/iteration goals
- Blockers and how they're being resolved
- Quality and testing status
- Upcoming milestones and key dates
- Risks and dependencies
- Any scope or timeline changes

## Release Phase — Communication Checklist

**When:** During release planning, testing, and deployment  
**Owner:** Communications/Operations Lead + Project Manager

### Personas to Involve
- [ ] **QA Lead** — Final testing and verification
- [ ] **Tech Lead** — Release readiness and technical sign-off
- [ ] **Developers** — Final bug fixes and deployment support
- [ ] **Security Officer** — Security scan results and sign-off
- [ ] **Communications/Operations Lead** — Release announcement and deployment coordination
- [ ] **Project Manager** — Release orchestration and timeline
- [ ] **Product Manager** — Release notes and customer messaging
- [ ] **Sponsor/Stakeholder** — Approval to proceed with release

### Communication Actions
- [ ] Share pre-release checklist and requirements
- [ ] Confirm all acceptance criteria met and tests passing
- [ ] Get security sign-off on vulnerability scans
- [ ] Draft and share release notes with Product Manager
- [ ] Announce release to support/customer success teams
- [ ] Send go/no-go decision to stakeholders
- [ ] Post-release verification communication
- [ ] Send release announcement to broader organization

### Key Messages
- What's included in the release
- Key features and improvements
- Bug fixes and known issues
- Deployment timeline and expected impact
- Migration steps or breaking changes (if any)
- Support and escalation contacts
- Post-release verification status

## Incident Response Phase — Communication Checklist

**When:** During security or operational incidents  
**Owner:** Communications/Operations Lead + Tech Lead

### Personas to Involve (Immediate)
- [ ] **Tech Lead** — Incident triage and diagnosis
- [ ] **Developers** — Technical remediation
- [ ] **Security Officer** — Security incident classification and containment
- [ ] **Communications/Operations Lead** — Incident communication and coordination
- [ ] **Project Manager** — Escalation to Sponsor if needed

### Personas to Involve (Ongoing)
- [ ] **QA Lead** — Regression testing and validation
- [ ] **Product Manager** — Customer impact assessment
- [ ] **Sponsor/Stakeholder** — High-level escalation (critical incidents only)

### Communication Actions
- [ ] Declare incident immediately via Slack/incident channel
- [ ] Send initial status update within 15 minutes (if customer-impacting)
- [ ] Establish war room / incident bridge
- [ ] Send updates every 30 minutes until resolved
- [ ] Notify customers of resolved incident
- [ ] Schedule post-incident blameless retrospective
- [ ] Share retrospective findings and action items

### Key Messages
- **Initial:** What's happening, who's working on it, expected timeline
- **Updates:** Progress, revised timeline, any new information
- **Resolution:** What was the issue, what we did to fix it, what we'll do to prevent it
- **Post-incident:** What we learned, action items, timeline for improvements

## Retrospective & Continuous Improvement Phase — Communication Checklist

**When:** After sprint, release, or major milestone  
**Owner:** Project Manager

### Personas to Involve
- [ ] **Developers** — What went well, what could improve
- [ ] **QA Lead** — Quality process feedback
- [ ] **Tech Lead** — Technical process feedback
- [ ] **Product Manager** — Product process feedback
- [ ] **Project Manager** — Execution process feedback
- [ ] **Sponsor/Stakeholder** — High-level feedback (optional, after major milestones)
- [ ] **Security Officer** — Security process feedback (if applicable)

### Communication Actions
- [ ] Schedule retrospective (45–75 min)
- [ ] Use anonymous idea board to encourage candor
- [ ] Prioritize 2–3 action items from retrospective
- [ ] Document action items with clear owners and due dates
- [ ] Share retrospective summary and action items with team
- [ ] Share high-level learnings with stakeholders
- [ ] Track action items in subsequent project syncs

### Key Messages
- What went well (celebrate wins)
- What could be improved (constructive feedback)
- Action items and owners
- Due dates and success criteria for improvements
- Learnings that can be applied to other teams

## Cross-Persona Collaboration Tips

### 1. **Keep It Simple & Actionable**
- Each persona should know exactly what's expected of them
- Use clear language, avoid jargon when possible
- Provide templates and checklists to reduce rework

### 2. **Use the Right Communication Channel**
- **Synchronous (meetings):** Decisions, complex discussions, alignment
- **Asynchronous (Slack, email, docs):** Updates, FYI, request for review
- **Broadcast (all-hands, dashboard):** Status, releases, major announcements

### 3. **Time-Box Communications**
- Standups: 15 minutes max
- Weekly syncs: 30–45 minutes max
- Retrospectives: 45–75 minutes
- Avoid constant back-to-back meetings; leave time for execution

### 4. **Respect Different Communication Preferences**
- Some personas prefer written updates (Developers, Tech Leads)
- Some prefer meetings (Sponsors, PMs)
- Some need both (QA, Security)
- Adapt your approach

### 5. **Use a Single Source of Truth**
- One project README for everyone
- One risk register updated weekly
- One project board with shared status
- Avoid duplicate information in multiple places

## Common Communication Anti-Patterns (What NOT to Do)

### ❌ "PM Sync Only" Approach
**Problem:** When only the Project Manager and Product Manager sync, gaps emerge with Developers, QA, and Tech Lead.  
**Solution:** Include all personas in weekly delivery sync. Keep it short (30–45 min) but ensure all voices are heard.

### ❌ "Surprise Escalation"
**Problem:** Escalating issues to Sponsor without warning or context.  
**Solution:** Communicate early and often. Surface risks in your regular updates so escalations are never a surprise.

### ❌ "Too Many Status Meetings"
**Problem:** Daily standups + weekly syncs + stakeholder syncs + retros = meeting overload.  
**Solution:** Consolidate where possible. Use asynchronous updates (project board, dashboard) to reduce live meeting time.

### ❌ "Silent QA"
**Problem:** QA Lead discovers blockers but doesn't surface them during standups.  
**Solution:** Make it safe to surface blockers. Emphasize "remove impediments" as a team goal, not blame.

### ❌ "One-Way Communication"
**Problem:** PM broadcasts updates but doesn't ask for feedback or input.  
**Solution:** Solicit feedback actively. Ask questions like "Any risks I'm missing?" or "Do we need to adjust?"

### ❌ "Release Night Surprises"
**Problem:** Security Officer discovers critical issue 1 hour before release.  
**Solution:** Shift left. Include Security in planning. Run security reviews during sprint, not at release gate.

## Weekly Status Update Template

**For:** Sponsors and key stakeholders  
**Frequency:** Weekly or as needed  
**Owner:** Project Manager or Communications/Operations Lead

```
Subject: [Project Name] Weekly Status — Week of [DATE]

**Status:** 🟢 ON TRACK / 🟡 AT RISK / 🔴 OFF TRACK

**Progress This Week:**
- Completed [X] items from backlog
- [Specific deliverable] is now ready for [next phase]
- [Key milestone] achieved
- Team velocity: [X] story points

**Next Steps:**
- Planning to start [upcoming feature/phase]
- Key milestones coming up: [dates]
- [Dependencies] needed from other teams

**Risks & Blockers:**
- [Risk 1]: Impact if not addressed
  - Mitigation: [action plan]
- [Blocker 1]: What's blocking progress
  - Escalation to [team] for [resolution]

**Quality & Testing:**
- Test coverage: [X]%
- Critical defects: [X]
- Release readiness: [status]

**Ask / Decisions Needed:**
- Need approval on [decision] by [date]
- Awaiting feedback on [item] from [person]
- Need resources for [activity]

**Contact:** [PM Name, Email, Phone]
```
